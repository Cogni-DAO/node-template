# x402 End-to-End: Hyperbolic Provider + Per-Request Settlement

> [!CRITICAL]
> This spec replaces the entire credit-purchase-then-consume billing model with x402 per-request settlement. The DB credit system (credit_ledger, payment_attempts, billing_accounts balance tracking) is deleted. LiteLLM remains the cost oracle. charge_receipts remain the audit trail.

## Context

The current billing architecture requires users to pre-purchase credits (USDC → on-chain verify → mint credits in DB), then consume credits per-request (LiteLLM → charge_receipts → debit credit_ledger). The outbound leg requires an operator wallet (Privy), Splits contract, and Coinbase Commerce integration to top up OpenRouter.

x402's `upto` scheme (shipped by Thirdweb, SDK v5.114.0+) enables a fundamentally simpler model: per-request dynamic settlement where the client authorizes a maximum amount and the server settles the actual cost after completion. Combined with Hyperbolic (x402 launch partner, open-source GPU inference), this enables **symmetric x402 on both legs** — no credit system, no pre-funding, no fiat bridge.

**Tradeoff:** Hyperbolic serves open-source models only (DeepSeek, Llama, Qwen, Mixtral). No Claude, GPT, or Gemini. This spec accepts that tradeoff for architectural simplicity and full crypto-native sovereignty.

## Goal

x402 inbound payment architecture: users/agents pay Cogni nodes per-request via x402 `upto` (USDC on Base). Nodes route to Hyperbolic via standard API key auth. LiteLLM is the cost oracle and model router (a per-node service, not a package). charge_receipts are the audit trail. No credit balances, no deposit flow, no operator wallet.

**P0 reality:** Inbound is x402. Outbound is API key. This is not asymmetric — it's phased. The outbound x402 path (node signs payments to Hyperbolic) is a P2 optimization that requires wallet signing infrastructure. P0 needs zero private keys.

## Non-Goals

| Item | Reason |
| - | - |
| Proprietary model access (Claude, GPT, Gemini) | Hyperbolic is open-source only. Hybrid OpenRouter fallback is a separate spec if needed. |
| COGNI utility token | x402 USDC per-request eliminates the need for a token on both legs. |
| Operator wallet (Privy) | No pre-funded provider balance needed — x402 settles per-request. |
| Splits contract | No revenue routing needed — DAO margin is computed and settled atomically per-request. |
| Coinbase Commerce integration | No fiat bridge — Hyperbolic accepts x402 USDC directly. |
| L402 (Lightning) support | P2+ consideration. This spec is x402-only. |
| Credit holds / pre-call reservations | x402 `upto` authorization IS the reservation. |

## Core Invariants

1. **X402_INBOUND**: Users/agents pay the node per-request via x402 `upto` (USDC on Base). The facilitator (Coinbase/Thirdweb hosted) handles on-chain settlement. **The node does NOT sign transactions for inbound payments — it only verifies and settles.** No private key needed.

2. **API_KEY_OUTBOUND**: Node pays Hyperbolic via standard API key auth (`HYPERBOLIC_API_KEY`). Same pattern as current OpenRouter. x402 outbound (node signing payments to Hyperbolic) is a P2 optimization, not P0.

3. **LITELLM_IS_THE_METER**: LiteLLM is a **per-node service** (Docker container) that does two things: (a) routes requests to Hyperbolic, (b) computes per-request cost via `x-litellm-response-cost` header or `usage.cost` streaming event. LiteLLM is the cost oracle. It is NOT a shared package — it is a running service every node operates independently.

4. **CHARGE_RECEIPTS_IMMUTABLE**: charge_receipts remain the immutable audit trail. Each receipt now includes `x402_settlement_tx` for on-chain proof. Idempotency via `UNIQUE(source_system, source_reference)` is unchanged.

5. **DAO_MARGIN_ATOMIC**: DAO margin is computed per-request: `user_settlement = provider_cost × MARKUP_FACTOR`. The margin (user_settlement - provider_cost) accumulates in the node's receiving wallet. No Splits contract needed.

6. **NO_CREDIT_BALANCE**: No DB-tracked credit balance. User's ability to pay is determined by their on-chain USDC balance + x402 authorization. Preflight is the x402 `verifyPayment()` call, not a DB balance check.

7. **RECEIVING_ADDRESS_NOT_SIGNING_KEY**: Each node has a receiving wallet address where x402 USDC settlements land. This address is configured, NOT a private key. The facilitator settles funds to this address. The node never signs outbound transactions in P0.

8. **OPEN_SOURCE_MODELS_ONLY**: This architecture supports only models served by Hyperbolic (open-source). No proprietary model fallback in this spec.

## Design

### 1. E2E Request Flow (P0 — API key outbound)

```
┌─────────────────────────────────────────────────────────────────────┐
│ INBOUND: User/Agent → Cogni Node (x402 upto)                       │
│ ──────────────────────────────────────                              │
│ 1. Agent sends POST /v1/chat/completions                            │
│ 2. Node reads model from request, looks up pricing in litellm config│
│ 3. Node computes maxPayableAmount:                                  │
│    = (input_tokens × input_cost) + (max_tokens × output_cost)       │
│      × USER_PRICE_MARKUP_FACTOR                                     │
│    (pricing.ts: calculateMaxPayable)                                │
│ 4. Node responds 402 with:                                          │
│    { scheme: "upto", maxPayableAmount, resource, chain: "base" }    │
│ 5. Agent signs upto authorization (USDC on Base)                    │
│ 6. Node calls verifyPayment() via facilitator                       │
│    — confirms authorization covers max                              │
│    — NO private key needed: facilitator verifies the agent's sig    │
│ 7. Proceed to INFERENCE                                             │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ INFERENCE: Cogni Node → Hyperbolic (standard API key)               │
│ ─────────────────────────────────────────────────────               │
│ 8. Node sends request to Hyperbolic via LiteLLM proxy               │
│    (LiteLLM is a per-node Docker service — the cost oracle)         │
│ 9. LiteLLM authenticates with HYPERBOLIC_API_KEY (standard bearer)  │
│10. Hyperbolic serves inference, streams response                    │
│11. LiteLLM captures response cost (x-litellm-response-cost header) │
│    This is the ACTUAL provider cost — the source of truth           │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ SETTLEMENT: Node settles with user (via facilitator)                │
│ ────────────────────────────────────────────────────                │
│12. Node computes user charge:                                       │
│    actual_user_cost = provider_cost × MARKUP_FACTOR                 │
│    (calculateLlmUserCharge — unchanged)                             │
│13. Node calls settlePayment(actual_user_cost) via facilitator       │
│    — Facilitator settles USDC to NODE_RECEIVING_ADDRESS             │
│    — NO private key: facilitator executes the authorized transfer   │
│14. charge_receipt written:                                          │
│    - source_reference = litellm_call_id (idempotent)                │
│    - x402_settlement_tx = facilitator settlement tx hash            │
│    - provider_cost_usd = raw Hyperbolic cost (from LiteLLM)        │
│    - response_cost_usd = user cost (with markup)                    │
│15. DAO margin = user_settlement - provider_cost                     │
│    Sits in node receiving wallet until epoch payout                 │
└─────────────────────────────────────────────────────────────────────┘
```

**Why this ordering?** The node must confirm the user can pay (step 6) before incurring provider cost (step 10). Actual settlement happens after completion (step 13) because the `upto` scheme settles actual cost, not max.

**Why no private key?** The node is the x402 SERVER, not the client. The agent (client) signs the payment authorization. The facilitator (Coinbase/Thirdweb hosted service) verifies the authorization and executes the on-chain transfer to the node's receiving address. The node only needs to call the facilitator API — no signing required.

**P2 evolution (x402 outbound):** When/if we want x402 outbound to Hyperbolic (eliminating the API key), the node becomes an x402 CLIENT and needs a signing wallet. That requires a `NodeWalletPort` abstraction (keystore/Vault/CDP), not a raw env var. This is a P2 concern.

### 3. x402 Middleware Architecture (P0 — Inbound Only)

New middleware intercepts requests before and after LiteLLM processing. Outbound to Hyperbolic is standard API key auth via LiteLLM — no x402 middleware on that leg.

**Inbound x402 middleware** (`src/adapters/server/x402/inbound-middleware.ts`):
- Intercepts `/v1/chat/completions` requests
- If no valid x402 payment header: respond 402 with `upto` payment requirements
- If valid payment header: call facilitator to verify, attach billing context, proceed to LiteLLM
- After LiteLLM completion: call facilitator to settle actual cost, write charge_receipt

**x402 facilitator adapter** (`src/adapters/server/x402/facilitator.ts`):
- Calls Thirdweb/Coinbase hosted facilitator API for `verifyPayment()` and `settlePayment()`
- Facilitator handles the on-chain USDC transfer — node never signs
- Settlement destination: `NODE_RECEIVING_ADDRESS`
- Can swap to self-hosted facilitator for sovereignty (P1)

**NOT in P0:**
- `outbound-client.ts` — No x402 outbound. LiteLLM calls Hyperbolic with `HYPERBOLIC_API_KEY` directly. No 402 negotiation on this leg.
- Node wallet signing — Node is receive-only in P0. No private keys.

### 4. Pricing Math (Unchanged Core, New Surface)

`calculateLlmUserCharge()` in `pricing.ts` is unchanged. The pipeline:

```
providerCostUsd (from LiteLLM)
  → userCostUsd = providerCostUsd × USER_PRICE_MARKUP_FACTOR
  → chargedCredits = ceil(userCostUsd × CREDITS_PER_USD)
```

New function for x402 max estimate:

```typescript
// pricing.ts — NEW
export function calculateMaxPayable(
  inputTokens: number,
  maxOutputTokens: number,
  inputCostPerToken: number,
  outputCostPerToken: number,
  markupFactor: number
): bigint {
  const maxProviderCost =
    inputTokens * inputCostPerToken + maxOutputTokens * outputCostPerToken;
  const maxUserCost = maxProviderCost * markupFactor;
  return usdToCredits(maxUserCost); // Same ceil rounding
}
```

### 5. DAO Margin Collection

No Splits contract. Margin is the spread between inbound x402 settlements and outbound provider cost:

```
Per request:
  User pays node (x402 settlement):  $0.0024 (provider cost × 2.0 markup)
  Node's Hyperbolic bill (API key):  $0.0012 (actual provider cost)
  Margin:                            $0.0012

Inbound USDC lands in NODE_RECEIVING_ADDRESS.
Outbound cost accrues on Hyperbolic account (paid from Hyperbolic pre-funded balance or billed).

Epoch payout:
  Margin from receiving wallet → DAO treasury → contributors (existing epoch settlement)
```

The epoch ledger system (activity_events, epoch_allocations, payout_statements) is unchanged. The `charge_receipts.provider_cost_usd` column enables accurate margin computation per-request.

**Note:** The Hyperbolic account must be funded (pre-paid balance or post-paid billing). This is analogous to the current OpenRouter balance, but simpler — no Privy, no Coinbase Commerce, no Splits. Just fund the Hyperbolic account.

## Open Questions

1. **Hyperbolic x402 uses `exact` scheme, not `upto`** — Hyperbolic's x402 endpoint (`hyperbolic-x402.vercel.app`, separate from `api.hyperbolic.xyz`) currently uses the core x402 `exact` scheme with EIP-3009 `transferWithAuthorization`. This means Hyperbolic sets a fixed per-request price, not dynamic per-token. The outbound leg is therefore `exact` (fixed price set by Hyperbolic) while the inbound leg is `upto` (dynamic, settled by us after LiteLLM computes actual cost). **This asymmetry exists at the protocol level, but both legs are x402 USDC on Base** — the difference is only in the scheme field. The x402 source code is at [github.com/HyperbolicLabs/hyperbolic-x402](https://github.com/HyperbolicLabs/hyperbolic-x402).

2. **Hyperbolic x402 endpoint is a Vercel wrapper** — The x402 path is a separate Vercel-deployed proxy, not the core Hyperbolic API. This introduces a dependency on that proxy's availability. Evaluate whether to use x402 outbound (wallet-gated, no API key) vs. standard API key auth (more reliable, established) for the outbound leg. Both options keep x402 on inbound.

3. **No embeddings on Hyperbolic** — Confirmed: no `/v1/embeddings` endpoint. text-embedding-3-small must continue through a separate provider. Options: OpenAI direct (requires fiat), Cohere (has crypto options), or self-hosted (e.g., BGE on Hugging Face). This is the one remaining provider dependency outside Hyperbolic.

4. **Hyperbolic account funding** — P0 uses API key auth to Hyperbolic, which means the Hyperbolic account needs a funded balance (pre-paid or post-paid billing). This is simpler than the Privy+Splits+Coinbase Commerce stack it replaces, but it's still a manual step. How is this funded? Options: (a) manual USDC transfer to Hyperbolic account, (b) credit card on Hyperbolic, (c) P2 x402 outbound eliminates this entirely.

5. **Max estimate UX** — The max payable amount can be 10-20x the actual cost (worst case: max_tokens fully used). Mitigation: Hyperbolic's unified per-token pricing makes estimates more predictable than split input/output pricing. Show estimated cost alongside max authorization.

6. **Hyperbolic rate limits** — Basic tier: 60 RPM. Pro: 600 RPM. May be insufficient for production agent workloads. Enterprise tier requires contacting Hyperbolic directly.
