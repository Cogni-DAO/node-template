## Binding Constraint

> **P06 — CI/CD primitives vs. product fit** (5/5) — Last evaluated: 2026-04-23 (reframed after Derek pushback)

**Reframe of earlier sub-eval.** The L02 scope-creep call missed the product. The repo is **designed** as a monorepo of N nodes (poly, resy, ai-only canary, +~50 future niches); each node is deliberately allowed to be "meaningless" at the product layer because the _real_ product is the **Operator** — an AI-driven git-manager that lets external AI contributors land PRs on DAO-owned node repos, eventually including private/external repos via a Cogni GitHub App. Poly is node #1 (proof-of-concept with real-money), resy is node #2, the unmerged "ai-only canary" PRs are the true proof-of-concept: can the Operator land PRs authored by AI contributors without human intervention?

**Binding constraint**: the CI/CD primitives currently in place were chosen for "ship one app to prod" and are structurally incompatible with "N node repos × many concurrent AI contributors". A top-0.1% team building this product would not ship the current stack. Specifically:

| Dimension          | Current                                                                                                                                                                                                                    | Top-0.1% target for this product                                                                                 |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| Build graph        | Per-app Docker + `detect-affected` scripts                                                                                                                                                                                 | Hermetic monorepo graph (Bazel / Turborepo / Nx) with content-addressed cache across nodes                       |
| Image promotion    | **Argo CD Image Updater** writing back to main ([Red Hat: Technology Preview, not production-recommended](https://docs.redhat.com/en/documentation/red_hat_openshift_gitops/1.19/html/release_notes/gitops-release-notes)) | Deterministic digest pins via Renovate-style PR bot — no branch-write loop                                       |
| Preview envs       | Single-slot `candidate-a` lease (`60dd98ba5 unstick: release candidate-a lease`)                                                                                                                                           | Per-PR ephemeral preview (Vercel / Fly / Render / ArgoCD ApplicationSets with PR labels) — unbounded concurrency |
| PR merging         | None                                                                                                                                                                                                                       | GitHub Merge Queue / Graphite / Aviator — mandatory at AI-PR volume                                              |
| Release control    | `workflow_run` chaining (multiple SHA-mismatch bugs: 0295/0343/0361)                                                                                                                                                       | Temporal-based release control plane (task.0290 planned — ship it)                                               |
| Per-node isolation | Bespoke kustomize + per-app migrator Dockerfiles                                                                                                                                                                           | Argo ApplicationSets + Helm chart template, one manifest per node                                                |
| Policy gates       | Workflow-time logic                                                                                                                                                                                                        | OPA/Conftest policy-as-code at PR time                                                                           |

The `fix(cicd)` burn rate (~40% of commits since last eval) is not developer sloppiness — it's the correct symptom of two specific wrong-primitive choices: (a) Argo CD Image Updater as the write path, (b) single-slot `candidate-a` as the flight path. Both directly cap the "N × AI contributors" throughput target.

**Work on now:**

1. **Retire Argo CD Image Updater.** Replace with a Renovate-style PR bot that opens a digest-bump PR. This breaks the branch-write-loop and kills the bug.0344 class of failures. Treat task.0297 (`core__vcs_flight_candidate`) and task.0360 (`/api/v1/vcs/pr`) as the foundation: the Operator writes digest PRs, CI runs them, merge queue lands them.
2. **Replace single-slot candidate-a with per-PR ephemeral preview envs.** Argo CD ApplicationSets + PR-number-keyed namespaces is the canonical pattern. This removes the lease-contention serialization point and unblocks the ai-only canary PRs that are the actual proof-of-concept.
3. **Ship task.0290 (Temporal release control plane).** It's already identified in MEMORY as the replacement for fragile `workflow_run` chaining — the current SHA-mismatch fires (bug.0295/0343/0361) are the case for doing it now, not after more nodes are added.
4. **Stand up a monorepo build graph** (Turborepo is the lowest-switching-cost option given Next.js baseline; Bazel if willing to accept the learning curve). Without this, each new node multiplies CI wall-time and compute cost.
5. **Enable GitHub Merge Queue** on main. Cheapest single lever for AI-PR-volume safety.
6. **Only after (1)–(3)**: treat L03 observability as the next unlock. AI contributors cannot self-verify without Loki/Langfuse/eval gates surfaced back into their PR comments.

This is also the honest answer to Derek's question: **no, the current CI/CD is not what a top-0.1% team would ship for this product.** It's what a solo operator ships for a single-node MVP, then tries to scale by adding nodes on top. The primitives need to be replaced, not hardened.
